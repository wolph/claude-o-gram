Build the session persistence layer and Fastify HTTP server that receives Claude Code hook POSTs.

Purpose: This is the "receiving end" -- Claude Code fires HTTP hooks to this server. The session store tracks which sessions are active and maps them to Telegram topics. The handlers manage session lifecycle (create, close, route) but delegate the Telegram actions to callbacks (wired in Plan 04).

  <name>Task 1: Implement session store with atomic JSON persistence</name>
  <files>src/sessions/session-store.ts</files>
  <action>
Create the SessionStore class that manages the in-memory session map and persists to disk.

1. Import `SessionInfo` from `../types/sessions.js` (note .js extension for ESM)
2. Import `writeFileSync, readFileSync, renameSync, existsSync, mkdirSync` from `node:fs`
3. Import `dirname` from `node:path`

4. Export `class SessionStore`:

   **Constructor:**
   - Takes `filePath: string` (path to sessions.json, e.g., `data/sessions.json`)
   - Initializes `private sessions = new Map<string, SessionInfo>()`
   - Calls `this.load()` to restore from disk on startup

   **Public methods:**

   `get(sessionId: string): SessionInfo | undefined`
   - Return from the Map

   `getByThreadId(threadId: number): SessionInfo | undefined`
   - Find by iterating values (small map, O(n) is fine for <100 sessions)

   `getActiveByCwd(cwd: string): SessionInfo | undefined`
   - Find an active session matching the given cwd
   - Used for session resume detection per Pitfall 1 from research: when a SessionStart with source "resume" arrives for a cwd that already has an active session, reuse existing topic

   `set(sessionId: string, info: SessionInfo): void`
   - Set in Map, call `this.save()`

   `updateStatus(sessionId: string, status: 'active' | 'closed'): void`
   - Update the status field, call `this.save()`

   `updateThreadId(sessionId: string, threadId: number): void`
   - Update threadId after Telegram topic creation, call `this.save()`

   `getActiveSessions(): SessionInfo[]`
   - Return all sessions with status === 'active'

   `incrementToolCount(sessionId: string): void`
   - Increment toolCallCount, call `this.save()`

   `addChangedFile(sessionId: string, filePath: string): void`
   - Add filePath to the session's filesChanged set
   - If the stored value is an array (loaded from JSON), convert to Set first
   - Call `this.save()`

   `getAllSessions(): SessionInfo[]`
   - Return Array.from(this.sessions.values())

   **Private methods:**

   `save(): void`
   - Serialize the Map to JSON. For each session, convert `filesChanged` from Set to Array for JSON serialization
   - Use atomic write: write to `filePath + '.tmp'`, then `renameSync` to filePath
   - Ensure parent directory exists with `mkdirSync(dirname(filePath), { recursive: true })`

   `load(): void`
   - If file does not exist, return (empty map is fine for first run)
   - Parse JSON, populate the Map
   - Convert `filesChanged` arrays back to Sets for in-memory use
   - Wrap in try/catch: if file is corrupt, log warning and start with empty map

5. Important: Node.js is single-threaded, so Map operations are atomic within the event loop (no race condition per Pitfall 4 from research). The save() method serializes writes naturally since they are synchronous.

Create the Fastify server and handler functions for the three hook endpoints.

**src/hooks/handlers.ts:**

1. Define a callback interface that the handlers use to delegate Telegram actions (will be wired in Plan 04):
   ```typescript
   export interface HookCallbacks {
     onSessionStart(session: SessionInfo): Promise<void>;
     onSessionEnd(session: SessionInfo): Promise<void>;
     onToolUse(session: SessionInfo, payload: PostToolUsePayload): Promise<void>;
   }
   ```

2. Export `class HookHandlers`:

   **Constructor:**
   - Takes `sessionStore: SessionStore` and `callbacks: HookCallbacks`

   **handleSessionStart(payload: SessionStartPayload): Promise<void>**
   - Derive topic name from `payload.cwd`: use `path.basename(payload.cwd)` as the name
   - Per user decision on duplicates: if another active session exists for the same cwd, append start timestamp for disambiguation, e.g., "my-project (14:32)"
   - Per user decision on resume: if `payload.source === 'resume'`, check `sessionStore.getActiveByCwd(payload.cwd)`. If found, update its sessionId to the new one (handles session ID instability per Pitfall 1) and call `callbacks.onSessionStart` with isResume=true. If not found, create new session.
   - Create a `SessionInfo` object with status 'active', toolCallCount 0, empty filesChanged Set
   - Store in sessionStore via `sessionStore.set()`
   - Call `callbacks.onSessionStart(session)` -- this will create/reopen the Telegram topic (Plan 03/04)
   - Note: threadId will be 0 initially; Plan 04 wiring will set it after topic creation

   **handleSessionEnd(payload: SessionEndPayload): Promise<void>**
   - Look up session by `payload.session_id`
   - If not found, log warning and return (defensive)
   - Update status to 'closed' via `sessionStore.updateStatus()`
   - Call `callbacks.onSessionEnd(session)` -- this will close the Telegram topic and post final summary

   **handlePostToolUse(payload: PostToolUsePayload): Promise<void>**
   - Look up session by `payload.session_id`
   - If not found, log warning and return (defensive)
   - Increment tool call count via `sessionStore.incrementToolCount()`
   - If the tool modifies files (tool_name is 'Write', 'Edit', 'MultiEdit'), extract the file path from tool_input and call `sessionStore.addChangedFile()`
   - Call `callbacks.onToolUse(session, payload)` -- this will format and send the Telegram message

3. Export `createHookHandlers(sessionStore: SessionStore, callbacks: HookCallbacks): HookHandlers`

**src/hooks/server.ts:**

1. Export `async function createHookServer(config: AppConfig, handlers: HookHandlers): Promise<FastifyInstance>`:

   - Create Fastify instance with `logger: true` (uses pino by default)
   - Register three POST routes:

   `POST /hooks/session-start`
   - Body is `SessionStartPayload`
   - Call `handlers.handleSessionStart(request.body)`
   - Return `{}` immediately (200 with empty JSON = success, per research Pattern 1)
   - Wrap in try/catch: log errors, still return 200 (do not block Claude Code even if handler fails)

   `POST /hooks/session-end`
   - Body is `SessionEndPayload`
   - Call `handlers.handleSessionEnd(request.body)`
   - Return `{}`
   - Same error handling

   `POST /hooks/post-tool-use`
   - Body is `PostToolUsePayload`
   - Call `handlers.handlePostToolUse(request.body)` -- fire and forget (do not await). Per Pitfall 6, respond immediately with 200 to avoid slowing down Claude Code.
   - Return `{}`

   `GET /health`
   - Return `{ status: 'ok', sessions: sessionStore.getActiveSessions().length }`
   - Useful for debugging

2. Do NOT call `fastify.listen()` here -- that will be done in the entry point (Plan 04). Just return the configured Fastify instance.

3. Important per research: All hook handlers should respond immediately with 200 `{}`. Processing (especially Telegram API calls) happens asynchronously. This prevents Claude Code from being slowed down by the bot.
