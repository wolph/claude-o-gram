# Plan 11-01: Sticky Message Dedup — Reconnect + Conditional initMonitoring

## Objective

Eliminate duplicate pinned status messages on /clear and bot restart by adding a `reconnect()` method to `StatusMessage` and branching `initMonitoring()` to use it when `session.statusMessageId > 0`.

## Context

Currently, `initMonitoring()` always calls `statusMsg.initialize()`, which sends a new message and pins it. This creates duplicate pins when:
1. A user runs /clear (new session, same topic)
2. The bot restarts (sessions reconnect with existing topics)

The fix has three parts:
1. Add `StatusMessage.reconnect(messageId)` to adopt an existing message
2. Branch `initMonitoring()` on `session.statusMessageId`
3. Inherit `statusMessageId` in `handleClearDetected()` (filesystem /clear path)

## must_haves

- [ ] `StatusMessage.reconnect(existingMessageId)` method exists and sets `this.messageId` without sending or pinning
- [ ] `initMonitoring()` calls `reconnect()` when `session.statusMessageId > 0`, calls `initialize()` when 0
- [ ] After reconnect, an immediate `requestUpdate()` fires to refresh stale content
- [ ] `handleClearDetected()` inherits `statusMessageId` from old session / clearPending
- [ ] Brand new sessions (statusMessageId === 0) still use `initialize()` path unchanged
- [ ] TypeScript compiles cleanly: `npx tsc --noEmit`

<task id="11-01-01" title="Add reconnect() method to StatusMessage">

### What
Add a `reconnect(existingMessageId: number): Promise<number>` method to the `StatusMessage` class that adopts an existing Telegram message ID without sending a new message or pinning.

### Why
This is the core primitive. All dedup flows (on /clear and on restart) need a way to tell StatusMessage "use this existing message ID" instead of creating a new one.

### Actions

1. **Open** `src/monitoring/status-message.ts`

2. **Add** `reconnect()` method to `StatusMessage` class, after `initialize()` (around line 72):

```typescript
/**
 * Reconnect to an existing status message by ID.
 * Adopts the message without sending or pinning a new one.
 * The next requestUpdate() will edit this message in-place.
 *
 * Used on /clear (reuse prior message) and bot restart (adopt stored message).
 */
async reconnect(existingMessageId: number): Promise<number> {
  this.messageId = existingMessageId;
  this.lastUpdateTime = Date.now();
  // Don't set lastSentText — forces the first requestUpdate() to actually
  // send an edit, refreshing any stale content from the prior session.
  return this.messageId;
}
```

<task id="11-01-02" title="Wire initMonitoring and handleClearDetected for dedup">

### What
1. Branch `initMonitoring()` to call `reconnect()` when `session.statusMessageId > 0`, else `initialize()`
2. Fix `handleClearDetected()` to inherit `statusMessageId` from the old session / clearPending data

### Why
This connects the `reconnect()` primitive to all three flows: /clear (HTTP hook), /clear (filesystem), and bot restart. Without this wiring, the reconnect method exists but is never called.

### Actions

1. **Open** `src/index.ts`

2. **Modify** `initMonitoring()` (around line 398): Replace the existing `initialize()` block with a conditional:

**Replace** the block:
```typescript
    // Start async initialization (status message send + pin) then start watchers
    void statusMsg.initialize().then((statusMessageId) => {
      sessionStore.updateStatusMessageId(session.sessionId, statusMessageId);
      // Set default verbosity from config
      sessionStore.updateVerbosity(session.sessionId, config.defaultVerbosity);
    }).catch((err) => {
      console.warn('Failed to initialize status message:', err instanceof Error ? err.message : err);
    });
```

**With:**
```typescript
    // Start async initialization: reconnect to existing message or create new one
    if (session.statusMessageId > 0) {
      // Dedup: adopt existing pinned message (on /clear or bot restart)
      void statusMsg.reconnect(session.statusMessageId).then(() => {
        // Trigger immediate update to refresh stale content with reset counters
        const latestSession = sessionStore.get(session.sessionId);
        if (latestSession) {
          statusMsg.requestUpdate(buildStatusData(latestSession, 'Starting...'));
        }
      });
    } else {
      // Brand new session: send + pin a new status message
      void statusMsg.initialize().then((statusMessageId) => {
        sessionStore.updateStatusMessageId(session.sessionId, statusMessageId);
        // Set default verbosity from config
        sessionStore.updateVerbosity(session.sessionId, config.defaultVerbosity);
      }).catch((err) => {
        console.warn('Failed to initialize status message:', err instanceof Error ? err.message : err);
      });
    }
```

3. **Modify** `handleClearDetected()` (around line 232): In the `newSessionData` object, change `statusMessageId: 0` to inherit from old session or pending data:

**Replace:**
```typescript
      statusMessageId: 0,
```

**With:**
```typescript
      statusMessageId: oldSession?.statusMessageId ?? pending?.statusMessageId ?? 0,
```

## Risk Assessment

**Low risk.** Changes are minimal (one new method, one conditional, one field inheritance). The existing `flush()` content dedup and error handling provide safety nets. If the adopted message was deleted by the user, `editMessageText` fails gracefully with a console warning.
