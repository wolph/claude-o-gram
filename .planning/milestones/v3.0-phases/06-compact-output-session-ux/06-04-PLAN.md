<objective>
Fix the /clear lifecycle to work when Claude Code only fires SessionStart(source=clear) WITHOUT a preceding SessionEnd(reason=clear), which is the current behavior due to upstream bug anthropics/claude-code#6428.

Purpose: All 3 UAT failures (tests 9, 10, 11: separator missing, no fresh status, counters not reset) share this single root cause. The current code assumes SessionEnd fires before SessionStart on /clear to populate a `clearPending` bridge map. Since SessionEnd never fires, the bridge is empty and the entire /clear flow silently fails. The fix restructures both `handlers.ts` and `index.ts` to work with SessionStart(source=clear) alone by finding the ACTIVE session (not recently closed) and extracting threadId/statusMessageId directly from it.

Output: Working /clear lifecycle that posts separator, creates fresh pinned status, and resets counters -- all triggered by SessionStart(source=clear) alone.
</objective>

```typescript
if (payload.source === 'clear') {
  // Primary path: find the ACTIVE session for this cwd.
  // Claude Code bug #6428: SessionEnd does NOT fire on /clear,
  // so the session is still active when SessionStart(source=clear) arrives.

  // already closed the session before this fires.
  const prior = existing ?? this.sessionStore.getRecentlyClosedByCwd(payload.cwd);

  if (prior) {
    // Reuse the topic: create new session entry with old threadId
    const clearNow = new Date().toISOString();
    this.sessionStore.set(payload.session_id, {
      ...prior,
      sessionId: payload.session_id,
      transcriptPath: payload.transcript_path,
      status: 'active',
      startedAt: clearNow,
      lastActivityAt: clearNow,
      toolCallCount: 0,
      filesChanged: new Set<string>(),
      suppressedCounts: {},
      contextPercent: 0,
      permissionMode: payload.permission_mode ?? prior.permissionMode,
    });
    await this.callbacks.onSessionStart(
      this.sessionStore.get(payload.session_id)!,
      'clear'
    );
    return;
  }
  // Fallback: if no session found at all, treat as new session
  // (handles edge cases where the bot restarted between clear events)
}
```

The callback `onSessionStart` then receives a session with the correct `threadId` and `statusMessageId` already set. Do NOT change the spread behavior.

```typescript
if (source === 'clear') {
  // /clear transition: reuse existing topic
  // The session object already has threadId and statusMessageId
  // from the prior session (copied via spread in handlers.ts).
  //
  // Two paths get us here:
  // 1. SessionEnd never fired (upstream bug #6428) -- prior session was active,
  //    handlers.ts found it via getActiveByCwd. threadId/statusMessageId are
  //    directly on the session object.
  // 2. SessionEnd fired first (future fix) -- clearPending was populated.
  //    handlers.ts found the closed session via getRecentlyClosedByCwd.
  //    threadId/statusMessageId are also on the session object.
  //
  // Either way, session.threadId and session.statusMessageId are valid.

  // Also check clearPending as an additional source (for future-proofing
  // when SessionEnd fires first and has already cleaned up the old session).
  const pending = clearPending.get(session.cwd);
  if (pending) {
    clearPending.delete(session.cwd);
  }

  // Clean up OLD monitoring for the previous session ID.
  // Since SessionEnd may not have fired, the old monitor may still be running.
  // Find it by iterating monitors for one matching this threadId.
  for (const [oldSessionId, monitor] of monitors) {
    if (oldSessionId !== session.sessionId && monitor.statusMessage) {
      // Check if this monitor belongs to the same thread
      const oldSession = sessionStore.get(oldSessionId);
      if (oldSession && oldSession.cwd === session.cwd) {
        monitor.transcriptWatcher.stop();
        monitor.summaryTimer.stop();
        monitor.statusMessage.destroy();
        monitors.delete(oldSessionId);

        // Also clean up control state for the old session
        approvalManager.cleanupSession(oldSessionId);
        inputRouter.cleanup(oldSessionId);

        // Mark old session as closed since SessionEnd never did it
        sessionStore.updateStatus(oldSessionId, 'closed');
        break;
      }
    }
  }

  // Post timestamped separator (SESS-02)
  const now = new Date();
  const timeStr = `${String(now.getHours()).padStart(2, '0')}:${String(now.getMinutes()).padStart(2, '0')}`;
  const separator = `\u2500\u2500\u2500 context cleared at ${timeStr} \u2500\u2500\u2500`;
  await batcher.enqueueImmediate(session.threadId, separator);

  // Reset session counters (SESS-05)
  sessionStore.resetCounters(session.sessionId);

  // Initialize fresh monitoring (creates new StatusMessage, pins it) (SESS-03)
  initMonitoring(session);

  // Re-detect input method
  void inputRouter.register(session.sessionId, session.cwd, session.permissionMode).then(() => {
    const method = inputRouter.getMethod(session.sessionId);
    if (method) {
      sessionStore.updateInputMethod(session.sessionId, method);
    }
  });

  return;
}
```

**Key changes from the current code:**

1. **Removed the `if (pending)` guard** around separator/reset. These now ALWAYS execute when source=clear, regardless of whether clearPending was populated.

2. **Added old monitor cleanup loop.** Since SessionEnd never fired, the old session's TranscriptWatcher, SummaryTimer, and StatusMessage are still running. The loop finds the old monitor by matching cwd (not sessionId, since the new session has a different ID) and cleans it up. This prevents resource leaks and ensures the old StatusMessage is destroyed before initMonitoring creates a new one.

3. **Removed unpin logic.** Per user feedback in UAT test 10: "no need to unpin the previously pinned status message." The old pinned message stays; the new one gets pinned alongside it (Telegram supports multiple pinned messages).

4. **clearPending is still consumed** if present (future-proofing), but is no longer REQUIRED for the flow to work.

5. **Old session marked as closed.** Since SessionEnd never fired, the old session is still 'active' in the store. We explicitly mark it 'closed' to prevent getActiveByCwd from finding stale sessions on subsequent /clear cycles.

**Do NOT modify** any other branches (resume, new session, onSessionEnd). The clearPending population in onSessionEnd remains as-is for future-proofing.
