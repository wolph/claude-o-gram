# v2.1 UX Overhaul Design

**Date:** 2026-03-01
**Milestone:** v2.1
**Goal:** Make Telegram output match Claude Code's terminal experience — compact, clean, and controllable.

## Features

### 1. Compact Tool Calls with Expand/Collapse

**Format:** Single-line, terminal-style:
```
Read(src/bot/bot.ts)
Edit(src/bot/formatter.ts) +3 -2
Bash(npm test)
Grep("pattern", src/**/*.ts)
Write(src/new-file.ts) 45 lines
```

**Truncation:** Max 250 chars per tool call. If over 250, truncate args with `…` and add `[Expand ▸]` inline button.

**Expand/Collapse behavior:**
- Tap Expand → `editMessageText()` replaces truncated text with full content, button becomes `[◂ Collapse]`
- Tap Collapse → reverts to truncated version
- Full content stored in `Map<string, string>` keyed by `chatId:messageId`
- Lost on bot restart (acceptable — messages are ephemeral)

**Batching:** Multiple tool calls within 2s window combine into one message, each on its own line. Expand/collapse applies to the entire message (expands all tool calls in that batch).

**File attachments:** Only for output exceeding 4096 chars after expansion (Telegram message limit). Everything else uses expand/collapse.

### 2. Terminal-Fidelity Output

**Remove all bot-generated commentary:**
- No "Claude is waiting for your input"
- No emoji-heavy status labels
- No bot action noise in the feed

**What stays:**
- Claude's actual text output (from transcript watcher) — posted as-is
- Tool calls in compact format
- Permission requests (real Claude Code events)
- Session lifecycle: topic creation, topic close

**Pinned status message simplified to:**
```
claude-o-gram | 45% ctx | 1h 30m
42 tools | 8 files
```
Compact single line, no emoji. Updated via `editMessageText()` with 3s debounce.

**Claude's text output:** Posted directly, preserving markdown where Telegram supports it. No wrapping in blockquotes or extra formatting.

### 3. /clear Reuses Topic

**Current:** `/clear` triggers `SessionStart(source: 'clear')` → creates new topic.

**New behavior:**
1. `handleSessionStart()` detects `source === 'clear'`
2. Looks up existing active session for same `cwd`
3. Reuses the same `threadId`
4. Posts separator: `─── context cleared ───`
5. Creates new `StatusMessage`, posts + pins it
6. Unpins old status message
7. Resets monitoring state (tool counts, timer, context %)

Topic name stays the same.

### 4. Permissions Visible Locally

The PreToolUse hook handler writes to stdout before blocking for Telegram response:
```
⚠ Permission: Bash(npm test) — waiting for Telegram approval...
✅ Approved by @user
```

`console.log()` in the hook handler before `waitForDecision()`, and again after the decision returns.

### 5. Permission Modes (No Timeout + Accept Modes)

**Remove 5-minute timeout.** `waitForDecision()` blocks indefinitely.

**Approval keyboard layout:**

Row 1: `✅ Approve` | `❌ Deny`
Row 2: `⚡ All` | `🔧 Same Tool` | `🛡️ Safe Only` | `⏳ Until Done`

**Mode behaviors:**

| Mode | Scope | Auto-clears when |
|------|-------|-----------------|
| Accept All | Every permission | User taps "Stop" |
| Same Tool | Same tool name only | User taps "Stop" |
| Safe Only | Green-risk tools only | User taps "Stop" |
| Until Done | Every permission | Claude's turn ends (Stop hook) |

**When a mode is activated:**
1. Immediately approve the current pending permission
2. Set `session.autoAcceptMode` on the session
3. Post sticky control message: `⚡ Auto-accepting: [mode]. Tap to stop.` with `🛑 Stop` button
4. Future permissions matching mode → auto-approve, post compact: `⚡ Tool(args...)`
5. Non-matching permissions (e.g., Bash under "Safe Only") → full approval prompt

**"Until Done" auto-clear:** Listen for `Stop` hook event. When received, clear mode and remove control message.

**Storage:** `autoAcceptMode` is session-level field in `SessionStore`. Cleared on session end, /clear, or manual stop.

### 6. Full Subagent Visibility

**Current:** `isSidechain` entries filtered out. Subagents invisible.

**New:** Process all transcript entries regardless of `isSidechain`.

**Subagent content format:**
```
▸ [researcher] Read(src/api/routes.ts)
▸ [researcher] Grep("pattern", src/)
▸ [researcher] Agent text output here...
```

**Lifecycle events (from Agent tool calls in main chain):**
```
▶ Agent(researcher) spawned — "Explore API patterns"
  type: general-purpose
```
```
✓ Agent(researcher) done
```

**Identity resolution:**
- `Agent` tool_use call in main chain contains `name`, `subagent_type`, `description`
- Match sidechain entries to parent Agent call
- Fallback: `[subagent]` if name unresolvable

**Nesting:** Prefix stacks for nested subagents: `▸▸ [inner-agent]`. Max 2 levels for readability.

## Architecture Notes

- Expand/collapse content map is in-memory only — acceptable for ephemeral messages
- Auto-accept modes are session-scoped, not global
- Subagent identity relies on transcript metadata — may need fallback heuristics
- All changes are backward-compatible with existing Telegram group setup
